# **Command (คอมมานด์ แพทเทิร์น)**

**Game Programming Patterns / Design Patterns Revisited**

Command (คอมมานด์) เป็นหนึ่งในแพทเทิร์นโปรดของผมเลยครับ โปรแกรมขนาดใหญ่ส่วนใหญ่ที่ผมเขียน ไม่ว่าจะเกมหรือโปรแกรมประเภทอื่น สุดท้ายแล้วมักจะได้นำมันมาใช้ในจุดใดจุดหนึ่งเสมอ เมื่อผมเลือกใช้มันในจุดที่ถูกต้อง มันจะช่วยแกะปมโค้ดที่พันกันยุ่งเหยิงได้อย่างหมดจด สำหรับแพทเทิร์นที่ยอดเยี่ยมขนาดนี้ กลุ่ม Gang of Four (GoF) กลับให้คำนิยามที่ยากจะเข้าใจตามคาดไว้ดังนี้ครับ:

> *จงแคปซูล (Encapsulate) คำร้องขอ (Request) ให้กลายเป็นวัตถุ (Object) เพื่อช่วยให้ผู้ใช้สามารถกำหนดค่าพารามิเตอร์ให้กับไคลเอนต์ (Client) ด้วยคำร้องขอที่แตกต่างกัน, สามารถนำคำร้องขอเข้าคิวหรือบันทึกประวัติ (Log) และรองรับปฏิบัติการที่สามารถกู้คืนกลับมาได้ (Undoable operations)*

ผมคิดว่าเราทุกคนคงเห็นพ้องตรงกันว่านั่นเป็นประโยคที่แย่มาก ข้อแรกเลยคือ มันทำลายภาพเปรียบเปรยที่ตัวมันเองพยายามจะสร้างขึ้น นอกเหนือจากโลกอันแปลกประหลาดของซอฟต์แวร์ที่คำศัพท์ต่างๆ จะมีความหมายเป็นอะไรก็ได้แล้ว คำว่า “ไคลเอนต์ (Client)” ในชีวิตจริงหมายถึง *มนุษย์* — ใครสักคนที่คุณทำธุรกิจด้วย และเท่าที่ผมจำได้ มนุษย์เราไม่สามารถ "ถูกกำหนดค่าพารามิเตอร์ (Parameterized)" ได้ครับ

จากนั้น ประโยคส่วนที่เหลือก็เป็นแค่การจดรายการสิ่งที่คุณ *อาจจะ* นำแพทเทิร์นนี้ไปใช้ประโยชน์ได้ ซึ่งมันไม่ได้ช่วยให้เห็นภาพชัดเจนขึ้นเลย เว้นแต่ว่าเคสที่คุณกำลังเจอมันจะไปตรงกับรายการเหล่านั้นพอดี คำจำกัดความแบบกระชับของผมสำหรับ Command pattern คือ:

> **Command คือการทำสายเรียกฟังก์ชัน (Method Call) ให้กลายเป็นวัตถุที่จับต้องได้ (Reified Method Call)**

*(โน้ต: คำว่า “Reify” มาจากภาษาละตินคำว่า “res” ที่แปลว่า “สิ่งของ” ผสมกับคำลงท้ายในภาษาอังกฤษว่า “–fy” ดังนั้นมันจึงมีความหมายตรงตัวว่า “ทำให้เป็นสิ่งของ” ซึ่งบอกตามตรงว่ามันฟังดูเป็นคำที่น่าสนุกกว่าเยอะเลยครับ)*

แน่นอนว่า คำว่า "กระชับ" บ่อยครั้งมักจะหมายถึง "สั้นจนอ่านไม่รู้เรื่อง" ดังนั้นคำนิยามนี้อาจจะไม่ได้ช่วยให้อะไรดีขึ้นเท่าไหร่ ขอผมแกะกล่องอธิบายมันเพิ่มอีกนิดครับ เผื่อคุณไม่เคยได้ยินคำว่า “Reify” มันแปลว่า “ทำให้กลายเป็นจริงขึ้นมา” หรืออีกคำหนึ่งที่มีความหมายเหมือนกันคือการทำให้สิ่งนั้นกลายเป็นวัตถุ “ระดับเฟิร์สคลาส (First-class)”

*(โน้ต: ระบบ Reflection ในบางภาษา ช่วยให้คุณสามารถทำงานกับพวก Types ในโปรแกรมของคุณได้โดยตรงในขณะที่โปรแกรมรันอยู่ (Runtime) คุณสามารถดึง Object ที่เป็นตัวแทนของ Class อื่นมาเล่นดูได้ว่า Type นั้นทำอะไรได้บ้าง พูดอีกอย่างหนึ่งก็คือ Reflection ก็คือระบบที่ทำให้ระบบ Type กลายเป็นวัตถุที่จับต้องได้นั่นเอง)*

ทั้งสองคำนี้หมายถึง **"การนำแนวคิด (Concept) บางอย่างมาแปลงให้กลายเป็นชิ้นส่วนข้อมูล — วัตถุ (Object) — ที่คุณสามารถยัดใส่ไว้ในตัวแปร หรือส่งผ่านเข้าไปในฟังก์ชันได้"** ดังนัั้น การที่ผมบอกว่า Command pattern คือ “Method call ที่กลายเป็นวัตถุ” สิ่งที่ผมหมายถึงก็คือ **มันคือการนำฟังก์ชันที่เราสั่งรัน มาห่อแรปไว้ข้างในวัตถุ (Object) ตัวหนึ่งนั่นเองครับ**

ฟังดูคล้ายๆ กับคำว่า “Callback”, “First-class function”, “Function pointer”, “Closure” หรือ “Partially applied function” ขึ้นอยู่กับว่าคุณมาจากสายภาษาไหน และในความเป็นจริงพวกมันก็อยู่ในละแวกเดียวกันนั่นแหละครับ กลุ่ม Gang of Four ได้กล่าวไว้ในภายหลังว่า:

> *Commands คือวัตถุสิ่งของที่เข้ามาทดแทนระบบ Callbacks ในรูปแบบของ Object-Oriented*

นี่ควรจะเป็นสโลแกนที่ดีกว่าประโยคแรกที่พวกเขาเลือกใช้ตั้งเยอะครับ

แต่ทั้งหมดนี้มันยังดูเป็นนามธรรมและเลือนลางเกินไป ผมชอบเปิดบทเรียนด้วยอะไรที่จับต้องได้จริง แต่ผมดันทำพังไปแล้วในย่อหน้าแรก เพื่อเป็นการชดเชย นับจากนี้เป็นต้นไปจะเป็นตัวอย่างสถานการณ์จริงที่ Command pattern จะเข้ามาทำหน้าที่ได้อย่างยอดเยี่ยมครับ

## **การตั้งค่าปุ่มกด (Configuring Input)**

ในเกมทุกเกม จะต้องมีโค้ดส่วนหนึ่งที่ทำหน้าที่ตรวจจับอินพุตดิบๆ จากผู้เล่น — ไม่ว่าจะเป็นการกดปุ่ม, เหตุการณ์บนคีย์บอร์ด, การคลิกเมาส์ หรืออะไรก็ตาม มันจะรับอินพุตเหล่านั้นเข้ามาแล้วแปลสภาพให้กลายเป็นการกระทำที่มีความหมายในเกม:

*(โน้ตจากผู้แปล: **[จุดนี้ในต้นฉบับมีรูปภาพ 1 รูป]** เป็นรูปผังแสดงการกดปุ่มบนจอยเกม Gamepad ชี้ลูกศรเชื่อมโยงไปยังฟังก์ชันคำสั่งในเกม เช่น กดปุ่ม X ชี้ไปที่ Jump(), กดปุ่ม Y ชี้ไปที่ FireGun())*

โค้ดที่ 구현 (Implement) แบบเรียบง่ายทื่อๆ จะหน้าตาประมาณนี้ครับ:

```cpp
void InputHandler::handleInput()
{
  if (isPressed(BUTTON_X)) jump();
  else if (isPressed(BUTTON_Y)) fireGun();
  else if (isPressed(BUTTON_A)) swapWeapon();
  else if (isPressed(BUTTON_B)) lurchIneffectively();
}

```

*(คำแนะนำโปร: อย่ากดปุ่ม B บ่อยนักล่ะ)*

ฟังก์ชันนี้มักจะถูกเรียกใช้งานเฟรมละหนึ่งครั้งโดย **Game Loop** และผมมั่นใจว่าคุณคงดูออกว่ามันทำหน้าที่อะไร โค้ดนี้ทำงานได้ดีครับตราบใดที่เรายินดีที่จะเชื่อมสายไฟปุ่มกดฝังตาย (Hard-wire) ไว้กับแอ็กชันในเกม แต่เกมยุคนี้ส่วนใหญ่จะเปิดโอกาสให้ผู้เล่นสามารถ **ตั้งค่าการเปลี่ยนปุ่มกด (Configure)** เองได้ตามใจชอบ

เพื่อรองรับฟีเจอร์นั้น เราจำเป็นต้องเปลี่ยนสายการเรียกฟังก์ชันตรงๆ อย่าง `jump()` และ `fireGun()` ให้กลายเป็นสิ่งที่เราสามารถ "สลับถอดเปลี่ยนได้" คำว่า "สลับถอดเปลี่ยนได้" ฟังดูคล้ายกับการกำหนดค่าให้กับตัวแปรเลยใช่ไหมครับ ดังนั้นเราจึงต้องการ **วัตถุ (Object)** ที่จะมาเป็นตัวแทนของแอ็กชันในเกม... และนี่คือจุดที่ Command pattern ก้าวเข้ามาครับ

เราจะเริ่มจากการประกาศ Base Class ที่เป็นตัวแทนของคำสั่งในเกมที่สามารถสั่งให้ทำงานได้:

```cpp
class Command
{
public:
  virtual ~Command() {}
  virtual void execute() = 0;
};

```

*(โน้ต: เมื่อคุณเจอ Interface ที่มีฟังก์ชันตัวเดียวโดดๆ และไม่มีการส่งคืนค่าใดๆ (Void) มีโอกาสสูงมากที่มันคือ Command pattern ครับ)*

จากนั้นเราก็สร้าง Subclasses ย่อยขึ้นมาสำหรับรองรับแต่ละแอ็กชันในเกม:

```cpp
class JumpCommand : public Command
{
public:
  virtual void execute() { jump(); }
};

class FireCommand : public Command
{
public:
  virtual void execute() { fireGun(); }
};

// บทอื่นๆ ก็ทำสไตล์เดียวกัน... คุณคงพอเห็นภาพแล้ว

```

ในคลาสจัดการอินพุตของเรา (Input Handler) เราจะเก็บตัวชี้ตำแหน่ง (Pointer) ของ Command ไว้สำหรับแต่ละปุ่มกด:

```cpp
class InputHandler
{
public:
  void handleInput();

  // ฟังก์ชันสำหรับผูกมัดคำสั่ง (Bind Commands)...
private:
  Command* buttonX_;
  Command* buttonY_;
  Command* buttonA_;
  Command* buttonB_;
};

```

คราวนี้ ตัวจัดการอินพุตก็จะทำหน้าที่แค่ส่งผ่านงาน (Delegate) ไปให้วัตถุเหล่านั้นรันแทน:

```cpp
void InputHandler::handleInput()
{
  if (isPressed(BUTTON_X)) buttonX_->execute();
  else if (isPressed(BUTTON_Y)) buttonY_->execute();
  else if (isPressed(BUTTON_A)) buttonA_->execute();
  else if (isPressed(BUTTON_B)) buttonB_->execute();
}

```

*(โน้ต: สังเกตไหมว่าเราไม่ได้เช็คค่า `NULL` ตรงนี้? เพราะโค้ดนี้ทึกทักเอาว่าทุกๆ ปุ่มกดจะมีคำสั่งบางอย่างผูกมัดไว้กับมันเสมอ)*

*(โน้ต: ถ้าเราอยากจะรองรับปุ่มกดที่ไม่ได้ทำอะไรเลย โดยที่ไม่ต้องมานั่งเขียนโค้ดเช็คค่า `NULL` ให้รกรุงรัง เราสามารถประกาศ Class ของ Command ตัวหนึ่งที่ฟังก์ชัน `execute()` ข้างในปล่อยว่างไว้เฉยๆ จากนั้น แทนที่จะเซ็ตค่าของปุ่มนั้นเป็น `NULL` เราก็แค่ชี้มันไปที่วัตถุว่างเปล่านั้นแทน แพทเทิร์นนี้มีชื่อเรียกว่า **Null Object Pattern** ครับ)*

จากเดิมที่อินพุตแต่ละตัวเคยพุ่งไปเรียกฟังก์ชันตรงๆ ตอนนี้เราได้เพิ่มชั้นของตัวกลาง (Layer of indirection) เข้ามาแทรกกลางแล้วครับ:

*(โน้ตจากผู้แปล: **[จุดนี้ในต้นฉบับมีรูปภาพ 1 รูป]** เป็นรูปผังแสดงสถาปัตยกรรมที่เปลี่ยนไป ปุ่มกดจอยเกมชี้ไปที่กล่องตรงกลางที่เป็น Command Object (เช่น JumpCommand) แล้วตัวกล่อง Command ค่อยชี้ต่อไปยังฟังก์ชันจริงอย่าง Jump())*

นี่คือ Command pattern ในแบบย่อส่วนครับ ถ้าคุณมองเห็นความดีงามของมันตั้งแต่ตอนนี้แล้ว ถือว่าเนื้อหาที่เหลือในบทนี้เป็นโบนัสแถมให้แล้วกันครับ!

### 🧠 **วิถี Senior Engineer: ถอดรหัสย่อหน้านี้สู่ Godot Architecture**

ในฐานะที่คุณกำลังก้าวไปเป็น Head Engineer ลองดูความเจ๋งของแนวคิดนี้ครับ:
ใน **Godot Engine** โครงสร้างตรงนี้ถูกแก้ปัญหามาให้แล้วในระดับสถาปัตยกรรมของ Engine ผ่านระบบที่เรียกว่า **Input Map** (ที่คุณไปตั้งค่าใน Project Settings ว่าปุ่มไหนคือ `"ui_accept"` หรือ `"jump"`)

แต่ถ้าคุณใช้ AI ในการเขียนโค้ดคุมตัวละคร โค้ดเดอร์ระดับ Mid-Level ทั่วไปมักจะสั่ง AI ว่า:

> *"ช่วยเขียนโค้ดถ้ากดปุ่ม jump ให้ตัวละครกระโดดที"* -> AI จะเจน `Input.is_action_just_pressed("jump")` ยัดเข้าไปในฟังก์ชัน `_process` ของตัวละครตัวนั้นโดยตรง ซึ่งเป็นการ **ผูกมัดแน่น (Tight Coupling)**

ถ้าคุณคิดแบบ **Senior / PM** คุณจะสั่งแยกส่วน (Decouple) มันออกมา เพื่อให้ระบบ Input สามารถรองรับ AI สั่งงาน หรือสลับตัวละครควบคุมได้ง่ายในอนาคตตามเนื้อหาหัวข้อถัดไปครับ!

## **คำสั่งสำหรับผู้แสดง (Directions for Actors)**

คลาส Command ต่างๆ ที่เราเพิ่งประกาศไปนั้นทำงานได้ดีในตัวอย่างก่อนหน้านี้ แต่มันยังมีข้อจำกัดค่อนข้างมากครับ ปัญหาคือพวกมันทึกทักไปเองว่ามีฟังก์ชันระดับบนสุด (Top-level) อย่าง `jump()` หรือ `fireGun()` ลอยอยู่ ซึ่งเป็นฟังก์ชันที่รู้ดีโดยปริยายว่าจะไปตามหาตัวละครของผู้เล่น (Player's avatar) เจอได้จากที่ไหน เพื่อบงการให้เขาเต้นระบำเหมือนกับหุ่นเชิด

การผูกมัดแน่น (Coupling) ที่แอบแฝงอยู่ตรงนั้นมันจำกัดความสามารถในการนำคำสั่งไปใช้งานต่อครับ เพราะกลายเป็นว่าคลาส `JumpCommand` จะสามารถสั่งให้ *ผู้เล่นเท่านั้น* ที่กระโดดได้ เรามาปลดล็อกข้อจำกัดนั้นกันดีกว่าครับ แทนที่จะเรียกฟังก์ชันที่วิ่งไปควานหาวัตถุด้วยตัวมันเอง เราจะทำการ **ส่งมอบวัตถุ (Pass in)** ที่เราต้องการจะบงการเข้าไปแทนเลยครับ:

```cpp
class Command
{
public:
  virtual ~Command() {}
  virtual void execute(GameActor& actor) = 0;
};

```

ตรงนี้ `GameActor` คือคลาส "วัตถุในเกม (Game Object)" ของเราที่เป็นตัวแทนของตัวละครในโลกของเกม เราส่งมันผ่านเข้าไปในฟังก์ชัน `execute()` เพื่อให้คำสั่งย่อยที่สืบทอดไปสามารถเรียกใช้ Method บนตัวละคร (Actor) ตัวไหนก็ได้ที่เราเลือกสรรมาครับ ดังนี้เลย:

```cpp
class JumpCommand : public Command
{
public:
  virtual void execute(GameActor& actor)
  {
    actor.jump();
  }
};

```

คราวนี้ เราสามารถใช้คลาสเดี่ยวๆ คลาสนี้คลาสเดียวเพื่อสั่งให้ตัวละคร *ตัวไหนก็ได้* ในเกมกระโดดโลดเต้นได้แล้วครับ ตอนนี้เราแค่ขาดชิ้นส่วนเชื่อมต่อระหว่างตัวจัดการอินพุต (Input Handler) กับคำสั่ง (Command) ที่จะคอยรับคำสั่งนั้นมาแล้วเอาไปสั่งรันบนวัตถุที่ถูกต้อง ข้อแรก เราต้องเปลี่ยนฟังก์ชัน `handleInput()` ให้มันทำการ **ส่งคืนค่า (Return)** ตัววัตถุ Command ออกมาแทนครับ:

```cpp
Command* InputHandler::handleInput()
{
  if (isPressed(BUTTON_X)) return buttonX_;
  if (isPressed(BUTTON_Y)) return buttonY_;
  if (isPressed(BUTTON_A)) return buttonA_;
  if (isPressed(BUTTON_B)) return buttonB_;

  // ไม่มีการกดปุ่มใดๆ ก็ไม่ต้องทำอะไร
  return NULL;
}

```

ตัวฟังก์ชันจัดการอินพุตมันไม่สามารถสั่งคำสั่งให้รันทันทีได้แล้ว เพราะมันไม่รู้ว่าจะต้องส่ง Actor ตัวไหนเข้าไป และนี่คือจุดที่เราได้ใช้ประโยชน์จากการที่ Command กลายเป็นวัตถุที่จับต้องได้ (Reified Call) — **เราสามารถชะลอเวลา (Delay) ในการสั่งรันคำสั่งออกไปตอนไหนก็ได้ตามใจชอบครับ**

จากนั้น เราต้องการโค้ดบางส่วนที่คอยรับคำสั่งนั้นมาแล้วจับมันไปรันบนตัวละครที่เป็นตัวแทนของผู้เล่น โค้ดจะหน้าตาประมาณนี้:

```cpp
Command* command = inputHandler.handleInput();
if (command)
{
  command->execute(actor);
}

```

หากทึกทักว่า `actor` คือตัวแปร Reference ที่อ้างอิงถึงตัวละครของผู้เล่น โค้ดนี้จะขับเคลื่อนตัวละครได้อย่างถูกต้องแม่นยำตามอินพุตของผู้ใช้ ซึ่งผลลัพธ์พฤติกรรมจะกลับมาเหมือนเป๊ะกับตัวอย่างแรกสุดที่เราเขียนไว้เลยครับ แต่ทว่า การเพิ่มชั้นตัวกลางคั่นกลางระหว่างคำสั่ง (Command) และตัวละคร (Actor) ผู้ลงมือทำ ได้มอบพลังความสามารถพิเศษที่เจ๋งสุดๆ มาให้เราหนึ่งอย่างครับนั่นคือ: **ตอนนี้เราสามารถปล่อยให้ผู้เล่นสลับไปควบคุมตัวละครตัวไหนก็ได้ในเกมแล้วครับ เพียงแค่เราเปลี่ยนตัวแปร Actor ที่เราส่งเข้าไปรันคำสั่งเท่านั้นเอง**

ในชีวิตจริง ฟีเจอร์สลับตัวละครควบคุมอาจจะไม่ได้เจอบ่อยในทุกเกม แต่มีกรณีตัวอย่างทำนองเดียวกันที่ **เกิดขึ้นบ่อยมากๆ** ครับ ที่ผ่านมาเราพิจารณาเฉพาะตัวละครที่ขับเคลื่อนโดยผู้เล่น (Player-driven) แล้วตัวละครตัวอื่นๆ ทั้งหมดในโลกของเกมล่ะ? พวกนั้นถูกขับเคลื่อนโดย AI ของเกมครับ เราสามารถใช้ Command pattern ตัวเดียวกันนี้แหละทำหน้าที่เป็น Interface เชื่อมต่อระหว่างเอนจิน AI กับตัวละครได้ โดยฝั่งโค้ด AI มีหน้าที่แค่ **ผลิตปล่อย (Emit) วัตถุ `Command` ออกมาเท่านั้นเองครับ**

การตัดความผูกมัด (Decoupling) ตรงนี้ ระหว่างฝั่ง AI ที่มีหน้าที่คัดเลือกคำสั่ง กับฝั่งตัวละครที่มีหน้าที่ลงมือปฏิบัติตามคำสั่ง มอบความยืดหยุ่นให้เรามหาศาลครับ เราสามารถใช้โมดูล AI ที่แตกต่างกันไปปลั๊กอินเข้ากับตัวละครแต่ละตัวได้ หรือจะผสมผสานจับคู่ AI เพื่อสร้างพฤติกรรมรูปแบบใหม่ๆ ก็ได้ อยากได้ศัตรูที่ดุร้ายขึ้นงั้นเหรอ? ก็แค่ปลั๊กอิน AI ตัวที่ก้าวร้าวเข้าไปทำหน้าที่ผลิตคำสั่งให้มันสิครับ ในความเป็นจริง เรายังสามารถเอา AI ชุดนี้ไปติดตั้งเชื่อมเข้ากับตัวละครของ *ผู้เล่น* ได้ด้วยซ้ำ ซึ่งจะมีประโยชน์มากในโหมดสาธิต (Demo mode) ที่ตัวเกมต้องรันระบบแบบขับเคลื่อนอัตโนมัติ (Auto-pilot)

ด้วยการ**ทำให้คำสั่งที่ใช้ควบคุมตัวละครกลายเป็นวัตถุระดับเฟิร์สคลาส (First-class objects)** เราได้ทำลายความผูกมัดที่แน่นหนาของการเรียกฟังก์ชันตรงๆ ทิ้งไปเรียบร้อยแล้ว ให้คุณจินตนาการภาพมันว่ามันกลายเป็น "คิว (Queue)" หรือ "สายธารของคำสั่ง (Stream of commands)" แทนครับ:

*(โน้ตจากผู้แปล: **[จุดนี้ในต้นฉบับมีรูปภาพ 1 รูป]** เป็นรูปท่อทรงกระบอกยาวๆ ที่มีกล่องคำสั่ง Command ไหลเรียงกันอยู่ข้างใน ด้านฝั่งซ้ายของท่อเขียนว่า Input/AI ทำหน้าที่ผลักคำสั่งเข้าท่อ ด้านฝั่งขวาเขียนว่า Actor คอยดึงคำสั่งไปใช้งาน)*

โค้ดฝั่งหนึ่ง (Input Handler หรือ AI) จะทำหน้าที่ **ผลิต (Produce)** คำสั่งแล้วนำมันไปวางไว้ในสายธาร โค้ดอีกฝั่งหนึ่ง (Dispatcher หรือตัว Actor เอง) จะทำหน้าที่ **บริโภค (Consume)** คำสั่งเหล่านั้นแล้วสั่งรันมัน การแปะคิวคั่นไว้ตรงกลางแบบนี้ ทำให้เราสามารถแยกฝั่งผู้ผลิต (Producer) ออกจากฝั่งผู้บริโภค (Consumer) ได้อย่างเด็ดขาดครับ

*(โน้ต: หากเรานำคำสั่งเหล่านี้มาทำให้อยู่ในรูปแบบที่สามารถบันทึกและส่งต่อได้ (**Serializable**) เราจะสามารถส่งสายธารคำสั่งเหล่านี้วิ่งผ่านเน็ตเวิร์กได้เลยครับ เราสามารถรับอินพุตของผู้เล่นจากเครื่องหนึ่ง ผลักมันข้ามระบบเครือข่ายไปซ่อมแซมรันบนอีกเครื่องหนึ่งได้ ซึ่งนี่คือจิ๊กซอว์ชิ้นสำคัญชิ้นหนึ่งในการทำเกมมัลติเพลเยอร์ออนไลน์ (Networked Multiplayer Game) ครับ)*

## **ระบบกู้คืนและทำซ้ำ (Undo and Redo)**

ตัวอย่างสุดท้ายนี้คือกรณีการใช้งานที่โด่งดังและเป็นที่รู้จักมากที่สุดของแพทเทิร์นนี้ครับ ถ้าวัตถุคำสั่งมันสามารถ **ลงมือทำ (Do)** สิ่งต่างๆ ได้ มันก็เป็นก้าวเล็กๆ อีกก้าวเดียวเท่านั้นที่มันจะสามารถ **กู้คืนระบบกลับมา (Undo)** ได้เช่นกัน ฟีเจอร์ Undo มักถูกใช้ในเกมแนววางแผนกลยุทธ์ (Strategy games) เพื่อให้ผู้เล่นสามารถย้อนกลับตาเดินที่ตัวเองไม่ชอบได้ และมันเป็นสิ่งที่ **จำเป็นต้องมี (De rigueur)** ในโปรแกรมเครื่องมือต่างๆ (Tools) ที่เราใช้สร้างเกม วิธีที่แน่นอนที่สุดที่จะทำให้ Game Designer ในทีมเกลียดขี้หน้าคุณ คือการส่งมอบโปรแกรมแก้ไขด่าน (Level Editor) ที่ไม่มีระบบ Undo ให้พวกเขาเวลาที่พวกเขาเผลอทำพลาดนิ้วเบียดครับ

*(คำเตือน: เรื่องนี้ผมพูดมาจากประสบการณ์ตรงเลยครับ)*

หากไม่มี Command pattern การทำระบบ Undo จะยากจนน่าประหลาดใจ แต่ถ้ามีมันอยู่ ทุกอย่างจะกลายเป็นเรื่องง่ายเหมือนปอกกล้วยเข้าปากเลยครับ สมมติว่าเรากำลังทำเกมเล่นคนเดียวแนวเทิร์นเบส (Turn-based game) และเราอยากให้ผู้เล่นกดกู้คืนตาเดินได้ เพื่อให้พวกเขาโฟกัสกับเรื่องกลยุทธ์ได้เต็มที่โดยไม่ต้องมานั่งเดาสุ่มความเสี่ยง

นับเป็นความโชคดีที่เราใช้ระบบคลาสคำสั่งในการแปลงอินพุตอยู่แล้ว ดังนั้นทุกๆ การเคลื่อนไหวที่ผู้เล่นทำจึงถูกแคปซูลห่อหุ้มไว้ในวัตถุเหล่านั้นเรียบร้อยแล้ว ตัวอย่างเช่น คำสั่งย้ายยูนิตทหารอาจจะหน้าตาแบบนี้ครับ:

```cpp
class MoveUnitCommand : public Command
{
public:
  MoveUnitCommand(Unit* unit, int x, int y)
  : unit_(unit),
    x_(x),
    y_(y)
  {}

  virtual void execute()
  {
    unit_->moveTo(x_, y_);
  }
private:
  Unit* unit_;
  int x_, y_;
};

```

สังเกตไหมครับว่าคลาสนี้มีความแตกต่างจากคำสั่งในหัวข้อก่อนหน้านี้นิดหน่อย ในตัวอย่างที่แล้ว เราต้องการ **แยกส่วน (Abstract)** คำสั่งออกจากตัวละครที่มันจะเข้าไปแก้ไข แต่ในกรณีนี้ เราต้องการ **ผูกมัด (Bind)** คำสั่งนี้เข้ากับยูนิตทหารตัวที่กำลังถูกสั่งเคลื่อนย้ายโดยเจาะจงเลย วัตถุอินสแตนซ์ของคำสั่งนี้จะไม่ใช่คำสั่งทั่วไปประเภท “สั่งย้ายอะไรบางอย่าง” ที่คุณจะเอาไปแปะใช้ในบริบทไหนก็ได้ แต่มันคือ "เหตุการณ์การเดินที่เฉพาะเจาะจงและจับต้องได้จริงในลำดับเทิร์นของเกม" ครับ

จุดนี้เน้นย้ำให้เห็นถึงความหลากหลาย (Variation) ในวิธีการ 구현 (Implement) ของ Command pattern ครับ ในบางกรณี (เหมือนตัวอย่างสองตัวแรก) คำสั่งจะเป็นวัตถุที่สามารถนำกลับมาใช้ซ้ำได้ (Reusable object) ซึ่งเป็นตัวแทนของ *แอ็กชันที่สามารถเกิดขึ้นได้* ตัวจัดการอินพุตก่อนหน้านี้ของเราจะถือครองวัตถุคำสั่งไว้เพียงชิ้นเดียวแล้วเรียกใช้ฟังก์ชัน `execute()` ทุกครั้งที่ปุ่มตรงสายถูกกดลงไป

แต่ในกรณีนี้ คำสั่งจะมีความเฉพาะเจาะจงมากกว่า มันเป็นตัวแทนของ *สิ่งที่มีการลงมือทำ ณ จุดเวลาใดเวลาหนึ่งโดยเฉพาะ* ซึ่งหมายความว่าโค้ดฝั่งจัดการอินพุตจะทำการ **สร้างอินสแตนซ์ใหม่ขึ้นมา (Create instance)** ทุกครั้งที่ผู้เล่นเลือกที่จะเดินหมาก โค้ดจะหน้าตาประมาณนี้ครับ:

```cpp
Command* handleInput()
{
  Unit* unit = getSelectedUnit();

  if (isPressed(BUTTON_UP)) {
    // ย้ายยูนิตขึ้นไปข้างบน 1 ช่อง
    int destY = unit->y() - 1;
    return new MoveUnitCommand(unit, unit->x(), destY);
  }

  if (isPressed(BUTTON_DOWN)) {
    // ย้ายยูนิตลงมาข้างล่าง 1 ช่อง
    int destY = unit->y() + 1;
    return new MoveUnitCommand(unit, unit->x(), destY);
  }

  // ตาเดินอื่นๆ...

  return NULL;
}

```

*(โน้ต: แน่นอนว่าในภาษาที่ไม่มีระบบเคลียร์แรมอัตโนมัติ (Non-garbage-collected) อย่างภาษา C++ สิ่งนี้หมายความว่าโค้ดส่วนที่มีหน้าที่สั่งรันคำสั่ง จะต้องคอยรับผิดชอบเรื่องการสั่งล้าง Memory คืนระบบด้วยตัวเองเมื่อใช้มันเสร็จแล้วครับ)*

ความจริงที่ว่าคำสั่งเหล่านี้เป็นแบบ "ใช้ครั้งเดียวทิ้ง (One-use-only)" จะกลายมาเป็นข้อได้เปรียบของเราในอีกไม่กี่วินาทีข้างหน้าครับ เพื่อทำให้คำสั่งเหล่านี้สามารถกู้คืนกลับมาได้ เราจำเป็นต้องประกาศปฏิบัติการอีกหนึ่งตัวที่ทุกๆ คลาสคำสั่งย่อยจำเป็นต้อง 구현 (Implement) เพิ่มเติมลงไปครับ:

```cpp
class Command
{
public:
  virtual ~Command() {}
  virtual void execute() = 0;
  virtual void undo() = 0;
};

```

ฟังก์ชัน `undo()` จะทำหน้าที่ตรงกันข้ามอย่างสิ้นเชิงเพื่อย้อนคืนสถานะของเกม (Game state) ที่ถูกเปลี่ยนไปโดยฟังก์ชัน `execute()` ตัวที่จับคู่กัน และนี่คือโค้ดย้ายยูนิตตัวเดิมของเราที่เพิ่มเติมระบบรองรับการ Undo เข้าไปครับ:

```cpp
class MoveUnitCommand : public Command
{
public:
  MoveUnitCommand(Unit* unit, int x, int y)
  : unit_(unit),
    xBefore_(0),
    yBefore_(0),
    x_(x),
    y_(y)
  {}

  virtual void execute()
  {
    // บันทึกตำแหน่งเดิมของยูนิตทหารเอาไว้ก่อนที่จะทำการย้าย
    // เพื่อที่เราจะได้สามารถกู้คืนมันกลับมาได้ในภายหลัง
    xBefore_ = unit_->x();
    yBefore_ = unit_->y();

    unit_->moveTo(x_, y_);
  }

  virtual void undo()
  {
    unit_->moveTo(xBefore_, yBefore_);
  }
private:
  Unit* unit_;
  int xBefore_, yBefore_;
  int x_, y_;
};

```

สังเกตไหมครับว่าเราได้เพิ่ม **สถานะภายในตัวแปร (More State)** เข้าไปในคลาสเพิ่มขึ้น เมื่อยูนิตทหารเดินเคลื่อนที่ไปข้างหน้า มันจะลืมตำแหน่งเดิมที่มันเคยอยู่ทันที ถ้าเราอยากจะสามารถกู้คืนการเดินครั้งนั้นได้ เราจำเป็นต้องเป็นคนจดจำตำแหน่งก่อนหน้าของยูนิตนั้นไว้ด้วยตัวเอง ซึ่งนั่นคือหน้าที่ของตัวแปร `xBefore_` และ `yBefore_` ที่เราเพิ่มเข้ามาครับ

*(โน้ต: จุดนี้ฟังดูเหมือนเป็นพื้นที่ที่ควรนำ **Memento Pattern** มาประยุกต์ใช้ใช่ไหมครับ แต่จากประสบการณ์ส่วนตัวผมพบว่ามันทำงานร่วมกันได้ไม่ค่อยดีเท่าไหร่ เนื่องจากคำสั่งคอมมานด์มักจะเข้าไปแก้ไขสถานะเพียงแค่เศษเสี้ยวเล็กๆ ของวัตถุเท่านั้น การสั่งถ่ายรูปสแนปชอตข้อมูลส่วนที่เหลือทั้งหมดเก็บไว้จึงเป็นการสิ้นเปลือง Memory โดยใช่เหตุ มันประหยัดทรัพยากรกว่าเยอะถ้าเราจะเลือกบันทึกเฉพาะชิ้นส่วนข้อมูลที่เราเข้าไปแตะต้องเปลี่ยนค่าด้วยมือเราเองครับ)*

*(โน้ต: โครงสร้างข้อมูลแบบคงทนถาวร (**Persistent Data Structures**) ก็เป็นอีกหนึ่งออปชันทางเลือกครับ ในสถาปัตยกรรมแบบนี้ ทุกๆ การแก้ไขวัตถุจะส่งคืนวัตถุตัวใหม่แกะกล่องกลับมาเสมอ โดยปล่อยให้วัตถุดั้งเดิมคงอยู่โดยไม่มีการเปลี่ยนแปลงใดๆ ด้วยระบบซอฟต์แวร์ที่ชาญฉลาด วัตถุใหม่เหล่านี้จะแชร์ข้อมูลดิบร่วมกันกับวัตถุก่อนหน้า ดังนั้นมันจึงประหยัดทรัพยากรกว่าการโคลนนิ่งวัตถุทั้งก้อนออกมาตรงๆ เยอะมาก เมื่อใช้งาน Persistent Data Structure คำสั่งแต่ละอันจะเก็บแค่ Reference ตัวชี้ไปยังวัตถุก่อนหน้าที่จะโดนสั่งรัน และการ Undo ก็มีความหมายเพียงแค่การสั่งสลับหัวชี้กลับไปใช้ Object ตัวเก่าเท่านั้นเองครับ)*

เพื่อเปิดโอกาสให้ผู้เล่นสามารถกด Undo การย้ายหมากได้ เราจะทำการเก็บรวบรวมวัตถุคำสั่งล่าสุดที่พวกเขาเพิ่งสั่งรันเอาไว้ เมื่อพวกเขาตะบันกดปุ่ม `Control-Z` บนคีย์บอร์ด เราก็จะวิ่งไปเรียกฟังก์ชัน `undo()` ของคำสั่งนั้นขึ้นมาทำงานครับ (และถ้าพวกเขากด Undo ไปแล้วรอบหนึ่ง ปุ่มนั้นก็จะแปลงร่างกลายเป็นระบบ "ทำซ้ำ (Redo)" และเราก็แค่สั่งเรียกฟังก์ชัน `execute()` ของคำสั่งนั้นให้ทำงานซ้ำอีกครั้งหนึ่ง)

การทำระบบรองรับ Undo หลายๆ ชั้น (Multiple levels of undo) ก็ไม่ได้ยากไปกว่านี้เท่าไหร่ครับ แทนที่เราจะจดจำคำสั่งแค่ชิ้นล่าสุดชิ้นเดียว เราก็เปลี่ยนมาเก็บรวบรวมคำสั่งไว้เป็น **รายการลิสต์ (List of commands)** พร้อมกับเก็บตัวชี้อ้างอิงตำแหน่งปัจจุบันว่าอยู่ตรงไหน (Current pointer) เมื่อผู้เล่นส่งคำสั่งรันงานมาใหม่ เราก็นำมันไปต่อท้ายลิสต์แล้วขยับตัวชี้ Current ให้มาแปะที่คำสั่งล่าสุดนั้น

เมื่อผู้เล่นเลือกคำสั่ง “Undo” เราก็ทำการกู้คืนคำสั่งปัจจุบันตัวที่ Current ชี้อยู่ แล้วถอยหลังตัวชี้ Current Pointer กลับไปหนึ่งก้าว เมื่อพวกเขาเลือกคำสั่ง “Redo” เราก็เดินหน้าขยับตัวชี้ Pointer ไปข้างหน้าหนึ่งก้าวแล้วสั่งรันฟังก์ชัน Execute ของคำสั่งนั้น และถ้าหากพวกเขากดเลือกสั่งคำสั่งใหม่แกะกล่องหลังจากที่เคยได้กด Undo ถอยหลังไปแล้ว ข้อมูลคำสั่งทั้งหมดในลิสต์ที่นอนอยู่ถัดจากตัวชี้ Current ปัจจุบันจะถูกลบทิ้งไปทันทีครับ

ครั้งแรกที่ผมสามารถ 구현 (Implement) ระบบนี้ได้สำเร็จในโปรแกรมแก้ไขด่าน (Level Editor) ผมรู้สึกว่าตัวเองเป็นอัจฉริยะเลยล่ะครับ ผมทึ่งมากที่มันมีความตรงไปตรงมาและทำงานได้อย่างยอดเยี่ยมขนาดนี้ มันอาจจะต้องใช้ระเบียบวินัยในตัวเองค่อนข้างสูงในการกำกับควบคุมให้แน่ใจว่า **"ทุกๆ การแก้ไขเปลี่ยนแปลงข้อมูลในเกมจะต้องวิ่งผ่านวัตถุ Command เสมอ"** แต่ทันทีที่คุณทำตามกฎเหล็กนี้ได้สำเร็จ ส่วนที่เหลือของระบบจะเป็นเรื่องง่ายทันทีครับ

*(โน้ต: ระบบ Redo อาจจะไม่ใช่ฟีเจอร์ที่พบเจอได้ทั่วไปในหน้าต่างเกมการเล่นปกติ แต่ระบบ **รีเพลย์ย้อนดูภาพการเล่น (Re-play)** นั้นเจอได้บ่อยมากครับ วิธีการเขียนระบบรีเพลย์แบบซื่อๆ ทื่อๆ คือการบันทึกสถานะของเกมทั้งหมด (Entire game state) ไว้ในทุกๆ เฟรมเพื่อนำมาเปิดเล่นย้อนหลัง แต่วิธีนั้นจะแดกแรม Memory มหาศาลจนเครื่องระเบิดแน่นอนครับ)*

*(ในทางกลับกัน เกมจำนวนมากใช้วิธีบันทึกเฉพาะเซตของวัตถุคำสั่ง (Set of commands) ที่ยูนิตทุกตัวในเกมลงมือทำในแต่ละเฟรมเก็บไว้แทน เวลาที่จะสั่งเล่นภาพรีเพลย์ย้อนหลัง ตัวเอนจินเกมก็แค่รันระบบจำลองสถานการณ์ตามปกติ (Game simulation) แล้วหยิบเอาคำสั่งที่เคยอัดบันทึกไว้ล่วงหน้าเหล่านั้นส่งป้อนกลับเข้าไปรันตามสเต็ปเฟรมเดิมเป๊ะๆ เท่านั้นเองครับ)*

## **รูปแบบคลาสที่สวยหรู กับวิถีฟังก์ชันนอลที่ไร้สถานะ? (Classy and Dysfunctional?)**

ในตอนต้น ผมได้กล่าวไว้ว่าคำสั่งคอมมานด์มีความคล้ายคลึงกับ First-class functions หรือ Closures (ฟังก์ชันที่หิ้วสถานะตัวแปรภายนอกไปกับตัวมันได้) แต่ตัวอย่างทุกตัวที่ผมแสดงให้คุณดูที่ผ่านมากลับเป็นการประกาศโครงสร้าง Class ทั้งสิ้น ถ้าคุณเป็นคนที่คุ้นเคยกับการเขียนโปรแกรมแนวฟังก์ชันนอล (Functional Programming) คุณคงกำลังเกาหัวสงสัยว่าแล้วพวกร่างฟังก์ชันมันหายไปไหนหมด?

เหตุผลที่ผมเขียนโค้ดตัวอย่างออกมาในรูปแบบนี้ก็เพราะว่าภาษา C++ มีระบบรองรับระบบ First-class functions ค่อนข้างจำกัดครับ ตัวชี้ฟังก์ชัน (Function pointers) ก็เป็นแบบไร้สถานะเก็บข้อมูลตัวแปรแนบไปไม่ได้, โครงสร้าง Functors ก็มีความแปลกประหลาดและยังจำเป็นต้องเขียนคลาสครอบอยู่ดี ส่วนระบบ Lambdas ที่เพิ่มเข้ามาใน C++11 ก็มีความยุ่งยากในการใช้งานสูงมากเนื่องจากมนุษย์ต้องเข้าไปบริหารจัดการ Memory ล้างแรมด้วยตัวเอง

แต่นั่น **ไม่ได้** หมายความว่าคุณไม่ควรนำโครงสร้าง Function มาใช้งานสำหรับทำ Command pattern ในภาษาอื่นนะครับ! หากคุณโชคดีได้ทำงานอยู่บนภาษาที่มีระบบ Closures ที่แท้จริงให้ใช้งาน ขอแนะนำให้ใช้พวกมันลุยได้เลยครับ! ในบางแง่มุมแล้ว Command pattern ก็คือหนทางหนึ่งในการจำลองระบบ Closures ขึ้นมาใช้งานในภาษาที่ไม่มีฟีเจอร์นี้รองรับนั่นเอง

*(โน้ต: ที่ผมพูดว่า "ในบางแง่มุม" เพราะการเขียนโครงสร้าง Class หรือ Structure ขึ้นมาเพื่อรองรับคำสั่งโดยเจาะจง ก็ยังมีประโยชน์ของมันอยู่ดีแม้แต่ในภาษาที่มีระบบ Closures ครบถ้วนครับ ถ้าคำสั่งของคุณมีปฏิบัติการหลากหลายมิติที่จะต้องทำ (เช่น คำสั่งที่มีฟังก์ชัน Undo แนบไปด้วย) การพยายามบีบจับทุกอย่างยัดรวมลงไปในฟังก์ชันเดี่ยวๆ ตัวเดียวจะมีความน่าเกลียดและเขียนยากมาก)*

*(นอกจากนี้ การประกาศคลาสจริงที่มีฟิลด์ระบุตัวแปรชัดเจน ยังช่วยให้ผู้ที่มาอ่านโค้ดสามารถดูออกได้ง่ายทันทีว่าคำสั่งนี้บรรจุข้อมูลอะไรซ่อนอยู่ข้างในบ้าง ระบบ Closures เป็นวิธีที่สั้นกระชับและงดงามมากในการห่อสถานะตัวแปรแบบอัตโนมัติ แต่มันมีความออโต้มากเกินไปจนบางครั้งทำให้ยากที่จะมองเห็นจากภายนอกว่าตัวมันกำลังถือครองตัวแปรสถานะอะไรอยู่บ้างกันแน่)*

ตัวอย่างเช่น ถ้าเรากำลังสร้างเกมขึ้นมาด้วยภาษา **JavaScript** เราจะสามารถสร้างคำสั่งเคลื่อนย้ายยูนิตทหารขึ้นมาได้สั้นๆ แบบนี้เลยครับ:

```javascript
function makeMoveUnitCommand(unit, x, y) {
  // ฟังก์ชันที่ถูกส่งคืนกลับไปตรงนี้แหละ ทำหน้าที่เป็นวัตถุคำสั่ง (Command object):
  return function() {
    unit.moveTo(x, y);
  }
}

```

เราสามารถเพิ่มระบบรองรับการ Undo เข้าไปด้วยได้เช่นกัน โดยการใช้ประโยชน์จาก Closures เป็นคู่ฟังก์ชันแบบนี้ครับ:

```javascript
function makeMoveUnitCommand(unit, x, y) {
  var xBefore, yBefore;
  return {
    execute: function() {
      xBefore = unit.x();
      yBefore = unit.y();
      unit.moveTo(x, y);
    },
    undo: function() {
      unit.moveTo(xBefore, yBefore);
    }
  };
}

```

ถ้าคุณเป็นคนที่แฮปปี้และคุ้นมือกับสไตล์การเขียนโค้ดแนวฟังก์ชันนอล (Functional style) วิธีการทำสิ่งต่างๆ แบบนี้จะรู้สึกเป็นธรรมชาติมากครับ แต่ถ้าคุณไม่คุ้นเคย ผมหวังว่าบทเรียนในบทนี้จะช่วยเปิดโลกและนำทางคุณให้เห็นภาพกว้างขึ้นได้ไม่มากก็น้อย สำหรับตัวผมแล้ว ประโยชน์ใช้สอยของ Command pattern แสดงให้เห็นชัดเจนเลยว่ากระบวนทัศน์แนวคิดของ Functional Paradigm มีประสิทธิภาพสูงมากในการแก้ปัญหาหลายๆ อย่างในโลกซอฟต์แวร์ครับ

## **ดูเพิ่มเติม (See Also)**

* คุณอาจจะจบลงด้วยการมีคลาสคำสั่งย่อยๆ เกิดขึ้นมาเต็มโปรเจกต์ไปหมด เพื่อทำให้การเขียนคลาสเหล่านั้นง่ายและสะอาดขึ้น บ่อยครั้งมักจะเป็นประโยชน์มากหากเราประกาศคลาสฐานที่เป็นรูปธรรม (Concrete base class) ขึ้นมาตัวหนึ่งที่มีการเขียนฟังก์ชันระดับบน (High-level methods) ที่อำนวยความสะดวกเตรียมไว้ให้ เพื่อให้คำสั่งย่อยที่สืบทอดไปสามารถหยิบฟังก์ชันเหล่านั้นมาผสมผสานประกาศพฤติกรรมของตัวเองได้ทันที สถาปัตยกรรมแบบนั้นจะเปลี่ยนฟังก์ชัน `execute()` หลักของคำสั่งให้กลายเป็นแพทเทิร์นที่ชื่อว่า **Subclass Sandbox Pattern** ครับ
* ในตัวอย่างทั้งหมดของเรา เราเป็นคนระบุแบบเจาะจงชัดเจน (Explicitly) ว่าจะเลือกส่ง Actor ตัวไหนเข้าไปทำงานร่วมกับคำสั่ง แต่ในบางสถานการณ์ โดยเฉพาะอย่างยิ่งในกรณีที่สถาปัตยกรรมวัตถุในเกมของคุณมีการเรียงลำดับชั้นเป็นโครงสร้างต้นไม้ (Hierarchical Model) การระบุตัวผู้รับอาจจะไม่ตรงไปตรงมาขนาดนั้น วัตถุชิ้นหนึ่งอาจจะรับคำสั่งมาตอบสนอง หรือมันอาจจะตัดสินใจปัดความรับผิดชอบส่งทอดคำสั่งนั้นไปให้วัตถุลูกน้องที่เป็นผู้ใต้บังคับบัญชาทำแทนก็ได้ หากคุณวางระบบสไตล์นั้น คุณได้ผสมผสานเข้ากับแพทเทิร์นที่ชื่อว่า **Chain of Responsibility Pattern** เรียบร้อยแล้วครับ
* คำสั่งบางอันจะไม่มีการเก็บสถานะตัวแปรภายในเลย (Stateless) เป็นเพียงแค่ชิ้นส่วนพฤติกรรมบริสุทธิ์ลอยๆ เช่น คลาส `JumpCommand` ในตัวอย่างแรกสุด ในกรณีสไตล์นั้น การสร้างอินสแตนซ์วัตถุตัวใหม่ขึ้นมาหลายๆ ชิ้นจะทำให้สิ้นเปลืองแรม Memory โดยเปล่าประโยชน์ เนื่องจากทุกๆ ชิ้นทำงานเหมือนกันเป๊ะ 100% แพทเทิร์นที่ชื่อว่า **Flyweight Pattern** จะก้าวเข้ามาช่วยแก้ปัญหาเพื่อบริหารจัดการพื้นที่ Memory ในจุดนี้ครับ

1. **ระเบียบวินัยในระบบ Undo (Architectural Discipline):** ย่อหน้าที่ผู้เขียนบอกว่า *"มันต้องใช้ระเบียบวินัยสูงมากในการคุมให้ทุกอย่างวิ่งผ่าน Command"* นี่คือหน้าที่ของ **Head Engineer** ครับ เมื่อคุณสั่งงาน AI หรือคนในทีม คุณต้องออกกฎเหล็กในโปรเจกต์เลยว่า ห้ามเขียนฟังก์ชันย้ายตำแหน่งตัวละครหรือปรับแต่งด่านตรงๆ ที่หน้า UI แต่ต้องเขียนคลาสสะสมเข้าไปในระบบประวัติ (Command History Stack) เสมอ ไม่งั้นระบบกู้คืนจะพังทันที
2. **การแปลงแนวคิดสู่การ 구현 ใน Godot (C# / GDScript):** * ใน **GDScript:** ระบบฟังก์ชันปิดหุ้มสถานะ (Closures) ในภาษา JavaScript ช่วงท้ายบท สามารถประยุกต์ใช้ใน GDScript ได้ผ่านฟีเจอร์ที่เรียกว่า **Callable** หรือการทำ **Anonymous Functions (람ดา)** ครับ เช่น `var my_command = func(): actor.jump()`
* ใน **C# (Godot):** มักจะนิยมใช้รูปแบบ Class ชัดเจนตามที่หนังสือแนะนำ โดยประยุกต์เก็บไว้ในโครงสร้าง `List<ICommand>` หรือ `Stack<ICommand>` เพื่อทำระบบประวัติย้อนหลังในเกมแนว Turn-based ของคุณนั่นเองครับ
