
# 📌 Part 1: Recap สรุปแก่นสำคัญของ Command Pattern

หากจะสรุปประโยคทองคำของบทนี้ให้เข้าใจง่ายที่สุด มันคือคำว่า: **"เปลี่ยนพฤติกรรม (คำสั่ง/การเรียกฟังก์ชัน) ให้กลายเป็นชิ้นส่วนข้อมูล (วัตถุ Object) คั่นกลางระหว่างทาง"**

### 1. ปัญหาของการไม่ใช้ (Tight Coupling)

โปรแกรมเมอร์ทั่วไปจะเชื่อมสายไฟดิบๆ จากอินพุตพุ่งตรงไปหาแอ็กชัน เช่น ปุ่ม Spacebar ──> สั่ง `player.jump()` ตรงๆ ปัญหาคือระบบจะฝังตาย วันหลังอยากทำระบบ **Key Rebinding (เปลี่ยนปุ่มกด)**, อยากให้ **AI (ศัตรู) กระโดดท่าเดียวกัน** หรืออยากทำระบบ **Undo/Redo** จะทำได้ยากมากเพราะไม่มีชิ้นส่วนข้อมูลกลางทางให้ไปจับต้อง

### 2. วิธีแก้ด้วย Command Pattern (Decoupling)

เราสร้างกล่องวัตถุกลางทางขึ้นมาแรปคำสั่งนั้นไว้ เช่น คลาส `JumpCommand`

* **ฝั่งผลิต (Input Handler / AI Decision):** มีหน้าที่แค่ตรวจจับสิทธิ์แล้วโยนกล่อง `Command` ส่งต่ออกมา
* **ฝั่งบริโภค (Character/Actor):** มีหน้าที่รับกล่องนั้นมา แล้วส่งตัวเองเข้าไปรันฟังก์ชัน `execute(actor)` ข้างในกล่อง

### 3. ผลประโยชน์สูงสุดที่ได้นำไปใช้ต่อ

* **Key Rebinding:** แค่สลับวัตถุเปลี่ยนกล่องตัวแปรปุ่มกด
* **Undo / Redo:** ในกล่องคอมมานด์แบบ `UndoableCommand` จะถูกเพิ่มฟังก์ชัน `undo()` และกล่องจะทำการ **"แพ็คสถานะในอดีต"** (เช่น `old_position`) เอาไว้ด้วย เมื่อเราเก็บกล่องคำสั่งเหล่านี้เรียงตามเวลาลงใน **History Stack (Array)** เราก็แค่ควักกล่องล่าสุดออกมาสั่ง `undo()` ได้อย่างแม่นยำ
* **Network / Replay:** สามารถแปลงกล่องคำสั่งเหล่านี้เป็นข้อมูลส่งข้ามเน็ตเวิร์ก หรืออัดเก็บไว้รันซ้ำเพื่อทำระบบภาพรีเพลย์ย้อนดูการเล่นได้โดยไม่เปลืองแรม

# 🤖 Part 2: หัวข้อใหม่ — วิธีนําแนวคิด Command Pattern ไปสั่งงาน AI ในยุค Vibe Coding & Agentic AI

ในยุคที่เราไม่ได้นั่งคีย์โค้ดเองทุกบรรทัด แต่เราใช้การ **"พิมพ์สั่ง AI (Vibe Coding)"** หรือปล่อยให้ **"บอทอัจฉริยะ (Agentic AI)"** วางโครงสร้างระบบให้ ปัญหาที่เจอบ่อยที่สุดคือ **AI มักจะชอบเขียนโค้ดแบบสะเพร่า มักง่าย และผูกมัดแน่น (Monolithic/Tight Coupling)** เพราะมันโฟกัสแค่การทำให้ฟีเจอร์รันผ่านตรงหน้าทันที

ในฐานะ **Head Engineer / PM** หน้าที่ของคุณคือการกำกับทิศทาง (Guardrails) ไม่ให้ AI พาโปรเจกต์ไปสู่ทางตัน นี่คือไกด์ไลน์ **Hands-on แบบพิมพ์สั่งตามได้ทันที** ครับ:

## 1. วิธีแก้ปม "AI เขียนโค้ดฝังตาย" (Anti-Monolithic Prompt)

เวลาสั่งให้ AI (เช่น Cursor, Claude, หรือ ChatGPT) เขียนระบบแอ็กชันของตัวละคร **ห้ามสั่งแค่ฟีเจอร์ทื่อๆ** แต่ต้องสั่งบังคับเชิงสถาปัตยกรรม (Architectural Enforcement) ตั้งแต่แรก

### 💬 ตัวอย่าง Prompt สั่งงาน AI (สายสถาปัตยกรรม):

> *"ช่วยเขียนระบบ Combat และความสามารถ (Skills) ของตัวละครใน Godot 4 ด้วยภาษา [GDScript / C#] โดยมีเงื่อนไขว่า **ห้ามเขียน Logic ของสกิลฝังไว้ในคลาส Character ตรงๆ เด็ดขาด** >
> แต่ให้ใช้หลักการ **Command Pattern** แยกสถาปัตยกรรมออกเป็นอิสระต่อกัน (Decoupled):
> 1. สร้างคลาสฐาน `BaseSkillCommand` (extends RefCounted) มีฟังก์ชัน `execute(actor: CharacterBody2D)`
> 2. สร้างคลาสย่อยแยกไฟล์ เช่น `SlashSkillCommand` และ `HealSkillCommand`
> 3. ออกแบบให้ฝั่งตัวละครมีระบบคิวรับคำสั่งเหล่านี้ไปประมวลผลต่อ"*
> 
>

## 2. การสั่งงาน Agentic AI ทำระบบ "Undo/Redo" ให้เสถียร

เวลาสั่งให้ AI ทำระบบย้อนเวลากลับ หรือระบบหน้าต่างแก้ไขด่าน (Editor Tools) AI มักจะเผลอใช้ท่าแย่ๆ อย่างการโคลนนิ่งฉากเก็บไว้ดื้อๆ ซึ่งเปลืองทรัพยากรมาก ให้เรากำกับ AI ด้วย Mental Model ของ **Redux / Command History**

### 💬 ตัวอย่าง Prompt สั่งงาน AI (สาย Undo/Redo):

> *"ฉันต้องการสร้างระบบตกแต่งห้องย้ายเฟอร์นิเจอร์ในเกม และต้องการระบบ Undo/Redo หลายชั้น (Multi-level)
> จง 구현 (Implement) ระบบนี้โดยอิงตามแนวคิด **Command Pattern และ Redux State Management**:
> * ทุกครั้งที่เฟอร์นิเจอร์โดนย้าย ให้สร้างวัตถุ `MoveFurnitureCommand` ขึ้นมากลางทาง ภายในกล่องนี้ต้องบันทึก `furniture_id`, `old_position`, และ `new_position` ไว้
> * สร้างสคริปต์ `HistoryManager` ที่ทำหน้าที่เป็นเซนเตอร์กลาง ถือครอง `Array` เพื่อทำหน้าที่เป็น Stack บันทึกคำสั่งเรียงตามเวลา
> * ห้ามใช้วิธีถ่าย Snapshot ข้อมูลห้องทั้งหมดเก็บไว้ ให้ใช้การสั่ง `.undo()` และ `.execute()` ย้อนสถานะผ่านวัตถุคำสั่งแต่ละชิ้นเท่านั้น เพื่อประหยัด Memory"*
> 
>

## 3. ทริคขั้นสูงในยุค Vibe Coding: ออกกฎเหล็กไว้ใน `.cursorrules` หรือ System Prompt

หากคุณใช้โปรแกรมอย่าง Cursor Engine คุณสามารถเขียนข้อตกลงร่วม (Rules) ครอบโปรเจกต์ของคุณไว้ได้เลย เพื่อที่เวลาคุณบ่นลอยๆ (Vibe Coding) ว่า *"อยากได้สกิลใหม่"* เจ้า AI Agent มันจะวิ่งไปสร้างไฟล์แยกสเต็ปตามแบบแผนของ Command Pattern ให้คุณโดยอัตโนมัติ

### 📄 โค้ดสำหรับใส่ในไฟล์ `.cursorrules` (หน้าต่างกำหนดทิศทาง AI):

```markdown
# โปรเจกต์สถาปัตยกรรม: เกมแนว Cozy Noir / Detective
# กฎเหล็กในการเขียนโค้ด (Coding Standards):

1. ทุกครั้งที่มีการสร้าง "การกระทำของตัวละคร" (Actions, Skills, Moves) ห้ามเขียนฝังลงใน Player Script
2. ต้องอิงหลักการ Command Pattern เสมอ:
   - แปลงแอ็กชันให้กลายเป็น Object เปล่ากลางทางก่อนส่งต่อ
   - ตัวรับอินพุต (Input) และตัวประมวลผล (Actor) ต้องไม่รู้จักกันตรงๆ แต่ต้องสื่อสารผ่านวัตถุคำสั่งกลาง
3. หากระบบต้องการกลไกย้อนเวลา หรือระบบประวัติศาสตร์ ให้เขียนฟังก์ชัน `undo()` และบันทึกข้อมูลเข้าสู่ศูนย์กลาง `HistoryManager` เสมอ

```
