# Command (คอมมานด์ แพทเทิร์น)
## Game Programming Patterns / Design Patterns Revisited

Command (คอมมานด์) เป็นหนึ่งในแพทเทิร์นโปรดของผมเลยครับ โปรแกรมขนาดใหญ่ส่วนใหญ่ที่ผมเขียน ไม่ว่าจะเกมหรือโปรแกรมประเภทอื่น สุดท้ายแล้วมักจะได้นำมันมาใช้ในจุดใดจุดหนึ่งเสมอ เมื่อผมเลือกใช้มันในจุดที่ถูกต้อง มันจะช่วยแกะปมโค้ดที่พันกันยุ่งเหยิงได้อย่างหมดจด สำหรับแพทเทิร์นที่ยอดเยี่ยมขนาดนี้ กลุ่ม Gang of Four (GoF) กลับให้คำนิยามที่ยากจะเข้าใจตามคาดไว้ดังนี้ครับ:

จงแคปซูล (Encapsulate) คำร้องขอ (Request) ให้กลายเป็นวัตถุ (Object) เพื่อช่วยให้ผู้ใช้สามารถกำหนดค่าพารามิเตอร์ให้กับไคลเอนต์ (Client) ด้วยคำร้องขอที่แตกต่างกัน, สามารถนำคำร้องขอเข้าคิวหรือบันทึกประวัติ (Log) และรองรับปฏิบัติการที่สามารถกู้คืนกลับมาได้ (Undoable operations)

ผมคิดว่าเราทุกคนคงเห็นพ้องตรงกันว่านั่นเป็นประโยคที่แย่มาก ข้อแรกเลยคือ มันทำลายภาพเปรียบเปรยที่ตัวมันเองพยายามจะสร้างขึ้น นอกเหนือจากโลกอันแปลกประหลาดของซอฟต์แวร์ที่คำศัพท์ต่างๆ จะมีความหมายเป็นอะไรก็ได้แล้ว คำว่า “ไคลเอนต์ (Client)” ในชีวิตจริงหมายถึง มนุษย์ — ใครสักคนที่คุณทำธุรกิจด้วย และเท่าที่ผมจำได้ มนุษย์เราไม่สามารถ "ถูกกำหนดค่าพารามิเตอร์ (Parameterized)" ได้ครับ

จากนั้น ประโยคส่วนที่เหลือก็เป็นแค่การจดรายการสิ่งที่คุณ อาจจะ นำแพทเทิร์นนี้ไปใช้ประโยชน์ได้ ซึ่งมันไม่ได้ช่วยให้เห็นภาพชัดเจนขึ้นเลย เว้นแต่ว่าเคสที่คุณกำลังเจอมันจะไปตรงกับรายการเหล่านั้นพอดี คำจำกัดความแบบกระชับของผมสำหรับ Command pattern คือ:

Command คือการทำสายเรียกฟังก์ชันให้กลายเป็นวัตถุที่จับต้องได้ (A command is a reified method call.)

คำว่า “Reify” แปลว่า “ทำให้กลายเป็นจริงขึ้นมา” หรืออีกคำหนึ่งที่มีความหมายเหมือนกันคือการทำให้สิ่งนั้นกลายเป็นวัตถุ “ระดับเฟิร์สคลาส (First-class)” หมายถึง "การนำแนวคิด (Concept) หรือพฤติกรรมบางอย่างมาแปลงให้กลายเป็นชิ้นส่วนข้อมูล — วัตถุ (Object) — ที่คุณสามารถยัดใส่ไว้ในตัวแปร หรือส่งผ่านเข้าไปในฟังก์ชันได้" ฟังดูคล้ายๆ กับคำว่า “Callback” หรือ “Closure” และในความจริงกลุ่ม Gang of Four ได้กล่าวไว้ในภายหลังว่า Commands คือวัตถุสิ่งของที่เข้ามาทดแทนระบบ Callbacks ในรูปแบบของ Object-Oriented นี่เป็นสโลแกนที่ดีกว่าเยอะครับ เพื่อไม่ให้เป็นนามธรรมเกินไป เรามาดูตัวอย่างสถานการณ์จริงที่คอมมานด์จะเข้ามาทำหน้าที่ได้อย่างยอดเยี่ยมกันครับ

การตั้งค่าปุ่มกด (Configuring Input)
ในเกมทุกเกม จะต้องมีโค้ดส่วนหนึ่งที่ทำหน้าที่ตรวจจับอินพุตดิบๆ จากผู้เล่น — ไม่ว่าจะเป็นการกดปุ่ม, เหตุการณ์บนคีย์บอร์ด หรือจอยเกม มันจะรับอินพุตเหล่านั้นเข้ามาแล้วแปลสภาพให้กลายเป็นการกระทำที่มีความหมายในเกม:

โค้ดแบบเรียบง่ายทื่อๆ (Hard-code) มักจะหน้าตาประมาณนี้:

```python  
# วิธีแบบเก่าที่ผูกมัดแน่น (Tight Coupling) ไม่แนะนำ
func handle_input():
    if Input.is_action_just_pressed("btn_x"): jump()
    elif Input.is_action_just_pressed("btn_y"): fire_gun()
    elif Input.is_action_just_pressed("btn_a"): swap_weapon()
```

โค้ดนี้ทำงานได้ดีตราบใดที่เรายินดีที่จะเชื่อมสายไฟปุ่มกดฝังตายไว้กับแอ็กชันในเกม แต่เกมยุคนี้ส่วนใหญ่จะเปิดโอกาสให้ผู้เล่นสามารถ ตั้งค่าการเปลี่ยนปุ่มกด (Configure) เองได้ตามใจชอบ เพื่อรองรับฟีเจอร์นั้น เราจำเป็นต้องเปลี่ยนสายการเรียกฟังก์ชันตรงๆ ให้กลายเป็นวัตถุที่เราสามารถกำหนดค่าใส่ตัวแปรและสลับถอดเปลี่ยนได้... และนี่คือจุดที่ Command pattern ก้าวเข้ามาครับ

เราจะเริ่มจากการประกาศ Base Class หรือพิมพ์เขียวกลางที่เกี่ยวข้องกับคำสั่ง:

📄 ชื่อไฟล์: command.gd (หรือ Command.cs)
GDScript
# สไตล์ GDScript
```python
class_name Command
extends RefCounted

# ฟังก์ชันเสมือน (Virtual) คลาสลูกต้องมาเขียนเนื้อหาจริงทับ
func execute() -> void:
    pass

```

```csharp 
// สไตล์ C#
public abstract class Command
{
    public abstract void Execute();
}
```

(โน้ต: เมื่อคุณเจอ Interface หรือ Class ที่มีฟังก์ชันตัวเดียวโดดๆ และไม่มีการส่งคืนค่าใดๆ มีโอกาสสูงมากที่มันคือ Command pattern ครับ)

จากนั้นเราก็สร้าง Subclasses ย่อยขึ้นมาสำหรับรองรับแต่ละแอ็กชันในเกม:

📄 ชื่อไฟล์: jump_command.gd (หรือ JumpCommand.cs)
GDScript
```python
class_name JumpCommand
extends Command

func execute() -> void:
    # เรียกฟังก์ชันกระโดด (ในที่นี้ทึกทักว่ามีฟังก์ชันระดับโลกหรือแชร์กันอยู่)
    global_jump_logic() 
```

📄 ชื่อไฟล์: fire_command.gd (หรือ FireCommand.cs)
GDScript
```python
class_name FireCommand
extends Command

func execute() -> void:
    global_fire_logic()
```
ในสคริปต์จัดการอินพุต เราจะประกาศตัวแปรเพื่อเก็บวัตถุคำสั่งเหล่านี้ไว้สำหรับแต่ละปุ่มกด คราวนี้ตัวจัดการอินพุตก็จะทำหน้าที่แค่ส่งผ่านงาน (Delegate) ไปให้วัตถุเหล่านั้นรันแทน:

📄 ชื่อไฟล์: input_handler.gd (หรือ InputHandler.cs)
GDScript
```python
class_name InputHandler
extends Node

# ตัวแปรสำหรับอุ้มวัตถุคำสั่ง (สลับเปลี่ยนได้ตลอดเวลาในหน้า Option)
var button_x_command: Command
var button_y_command: Command

func _ready() -> void:
    # จับคู่ปุ่มกดกับคลาสคำสั่งตอนเริ่มเกม
    button_x_command = JumpCommand.new()
    button_y_command = FireCommand.new()

func handle_input() -> void:
    if Input.is_action_just_pressed("btn_x") and button_x_command:
        button_x_command.execute()
    elif Input.is_action_just_pressed("btn_y") and button_y_command:
        button_y_command.execute()
```

จากเดิมที่อินพุตแต่ละตัวเคยพุ่งไปเรียกฟังก์ชันตรงๆ ตอนนี้เราได้เพิ่มชั้นของตัวกลาง (Layer of indirection) เข้ามาแทรกกลางเรียบร้อยแล้ว นี่คือ Command pattern ในแบบย่อส่วนครับ

คำสั่งสำหรับผู้แสดง (Directions for Actors)
คลาส Command ด้านบนยังมีความจำกัดอยู่ครับ เพราะมันทึกทักไปเองว่ามีฟังก์ชันอย่าง jump() ลอยอยู่ซึ่งรู้ดีโดยปริยายว่าจะไปหาตัวละครของผู้เล่นเจอได้จากที่ไหน การผูกมัดแน่น (Coupling) ตรงนั้นทำให้คำสั่งนี้ใช้ได้กับแค่ตัวผู้เล่นคนเดียวเท่านั้น

เรามาปลดล็อกข้อจำกัดนั้นกันดีกว่าครับ แทนที่จะเรียกฟังก์ชันที่วิ่งไปควานหาวัตถุด้วยตัวมันเอง เราจะทำการ ส่งมอบวัตถุ (Pass in) ที่เราต้องการจะบงการเข้าไปในฟังก์ชัน execute() แทนเลยครับ:

📄 ชื่อไฟล์: command.gd (เวอร์ชันอัปเกรด)
GDScript
```python
class_name Command
extends RefCounted

# รับตัวละคร (Actor) เข้ามาทางพารามิเตอร์ เพื่อสั่งงานให้ถูกตัว
func execute(actor: CharacterBody2D) -> void:
    pass
```

คราวนี้เรานำไปเขียนคลาสลูกให้รับช่วงต่ออย่างยืดหยุ่น:

📄 ชื่อไฟล์: jump_command.gd (เวอร์ชันอัปเกรด)
GDScript
```python
class_name JumpCommand
extends Command

func execute(actor: CharacterBody2D) -> void:
    if actor.has_method("jump"):
        actor.jump() # สั่งให้ Actor ตัวที่ส่งเข้ามาทำงานกระโดด
```

ตอนนี้เราสามารถใช้คลาสเดี่ยวๆ คลาสนี้คลาสเดียวเพื่อสั่งให้ตัวละคร ตัวไหนก็ได้ ในเกมกระโดดโลดเต้นได้แล้วครับ ชิ้นส่วนถัดไปคือเราจะเปลี่ยน handle_input() ให้ทำหน้าที่ ส่งคืนค่า (Return) วัตถุคำสั่งออกมาแทนการรันทันที เพราะตัวตรวจจับอินพุตไม่ควรมีหน้าที่ไปรับรู้ว่าต้องเอาคำสั่งนี้ไปสั่งรันบนตัวละครตัวไหน

📄 ชื่อไฟล์: input_handler.gd (เวอร์ชันอัปเกรด)
GDScript
```python
class_name InputHandler
extends Node

var button_x_command: Command
var button_y_command: Command

# คืนค่าวัตถุคำสั่งออกมากลางทาง เพื่อเอาไปใช้ประโยชน์ต่อในภายหลัง
func get_input_command() -> Command:
    if Input.is_action_just_pressed("btn_x"): return button_x_command
    if Input.is_action_just_pressed("btn_y"): return button_y_command
    return null
```
จากนั้น ในโค้ดควบคุมหลักของเกม (Game Loop) เราก็นำคำสั่งที่ได้มานั้นไปสั่งรันบนตัวละครที่เราต้องการบงการ:

GDScript
```python
# ตัวอย่างใน Game Loop หรือสคริปต์ควบคุมหลัก
var command: Command = input_handler.get_input_command()
if command:
    command.execute(player_character) # ส่ง Node ตัวละครผู้เล่นเข้าไป
```
การเพิ่มชั้นตัวกลางคั่นกลางระหว่างคำสั่งและตัวละครผู้ลงมือทำ ได้มอบพลังความสามารถพิเศษที่เจ๋งสุดๆ มาให้เราหนึ่งอย่างครับนั่นคือ: ตอนนี้เราสามารถปล่อยให้ผู้เล่นสลับไปควบคุมตัวละครตัวไหนก็ได้ในเกม หรือแม้กระทั่งปล่อยให้ AI ของเกมเป็นคนผลิตวัตถุ Command ส่งป้อนเข้ามาให้ตัวละครแทนอินพุตของมนุษย์ก็ได้

การตัดความผูกมัด (Decoupling) ตรงนี้ ระหว่างฝั่ง AI หรืออินพุตที่เป็นผู้ผลิตคำสั่ง กับฝั่งตัวละครที่เป็นผู้ลงมือปฏิบัติ ทำให้เกมมีโครงสร้างเหมือนระบบคิวหรือสายธารของคำสั่ง (Stream of commands) โค้ดฝั่งหนึ่งผลิตคำสั่ง โค้ดอีกฝั่งหนึ่งบริโภคคำสั่ง แปะคิวคั่นไว้ตรงกลาง หากเรานำคำสั่งเหล่านี้มาแปลงเป็นรูปแบบที่สามารถส่งต่อข้อมูลได้ (Serializable) เราจะสามารถส่งสายธารคำสั่งนี้วิ่งผ่านระบบเน็ตเวิร์กเพื่อทำเกมมัลติเพลเยอร์ออนไลน์ได้ทันทีครับ

ระบบกู้คืนและทำซ้ำ (Undo and Redo)
ตัวอย่างสุดท้ายนี้คือกรณีการใช้งานที่โด่งดังที่สุดครับ ถ้าวัตถุคำสั่งมันสามารถ ลงมือทำ (Do) สิ่งต่างๆ ได้ มันก็เป็นก้าวเล็กๆ อีกก้าวเดียวที่มันจะสามารถ กู้คืนระบบกลับมา (Undo) ได้เช่นกัน ฟีเจอร์ Undo เป็นสิ่งที่จำเป็นต้องมี (De rigueur) ในโปรแกรมเครื่องมือแก้ไขด่าน (Level Editor) วิธีที่แน่นอนที่สุดที่จะทำให้ Game Designer เกลียดขี้หน้าคุณ คือการส่งมอบโปรแกรมที่ไม่มีระบบ Undo ให้เวลาที่พวกเขาทำงานพลาดครับ

หากไม่มี Command pattern การทำระบบ Undo จะยากจนน่าประหลาดใจ แต่ถ้ามีมันอยู่ ทุกอย่างจะกลายเป็นเรื่องง่ายเหมือนปอกกล้วยเข้าปากเลยครับ สมมติว่าเรากำลังทำเกมแนวเทิร์นเบส (Turn-based game) ย้ายยูนิตทหาร และเราต้องการให้ผู้เล่นกู้คืนตาเดินได้

ในตอนที่สร้างวัตถุนี้ขึ้นมากลางทาง มันจะทำการ "แพ็คข้อมูล" ตัวละครยูนิตทหาร และตำแหน่งใหม่ที่จะเดินไปล็อกเก็บไว้ข้างในอินสแตนซ์วัตถุตัวนั้นทันที:

📄 ชื่อไฟล์: move_unit_command.gd (หรือ MoveUnitCommand.cs)
GDScript
```python
class_name MoveUnitCommand
extends Command

var _unit: Node2D      # แพ็คข้อมูล: ยูนิตตัวไหนเดิน
var _new_x: int        # แพ็คข้อมูล: ตำแหน่งเป้าหมาย X
var _new_y: int        # แพ็คข้อมูล: ตำแหน่งเป้าหมาย Y

var _old_x: int        # สถานะเพิ่มเติม: เอาไว้จดจำอดีตตอนกด Undo
var _old_y: int        # สถานะเพิ่มเติม: เอาไว้จดจำอดีตตอนกด Undo

func _init(unit: Node2D, x: int, y: int) -> void:
    _unit = unit
    _new_x = x
    _new_y = y

func execute(actor: CharacterBody2D) -> void:
    # ก่อนจะก้าวไปข้างหน้า ต้องจดจำตำแหน่งปัจจุบันเอาไว้ก่อนเสมอ
    _old_x = _unit.position.x
    _old_y = _unit.position.y
    
    # สั่งเคลื่อนที่ไปยังตำแหน่งใหม่
    _unit.move_to(_new_x, _new_y)
```
สังเกตนะครับว่าคลาสนี้มีความเฉพาะเจาะจงมาก มันทำหน้าที่เป็นตัวแทนของ สิ่งที่มีการลงมือทำ ณ จุดเวลาใดเวลาหนึ่งโดยเฉพาะ ซึ่งหมายความว่าโค้ดฝั่งจัดการอินพุตจะทำการ สร้างอินสแตนซ์วัตถุใหม่ขึ้นมา (Create instance) ทุกครั้งที่ผู้เล่นเลือกที่จะเดินหมาก เพื่อให้คำสั่งสามารถ Undo ได้ เราจะเพิ่มฟังก์ชัน undo() เข้าไปในพิมพ์เขียวหลัก:

📄 ชื่อไฟล์: command.gd (เวอร์ชันรองรับ Undo)
GDScript
```python
class_name Command
extends RefCounted

func execute(actor: CharacterBody2D) -> void: pass
func undo() -> void: pass # ย้อนคืนสถานะของเกมที่ถูกเปลี่ยนไป
```
และนี่คือฟังก์ชัน undo() ที่เพิ่มเข้าไปในคลาสดึงทหารกลับตำแหน่งเก่า:

GDScript
```python
# เพิ่มเข้าไปในไฟล์ move_unit_command.gd
func undo() -> void:
    # ย้ายยูนิตทหารกลับไปยังตำแหน่งอดีตที่เคยจดจำไว้ในกล่องวัตถุตัวนี้
    _unit.move_to(_old_x, _old_y)
```
เมื่อทหารขยับที่ มันจะลืมตำแหน่งเดิมทันที การที่เราแพ็คตัวแปรเก็บอดีต (_old_x, _old_y) ไว้ในกล่องคำสั่ง ทำให้เราไม่ต้องเสี่ยงเซฟข้อมูลสแนปชอตวัตถุทั้งก้อนให้เปลืองแรม (ซึ่งช่วยประหยัด Memory กว่าการใช้ Memento Pattern มากครับ)

เพื่อเปิดโอกาสให้ผู้เล่นสามารถกด Undo ได้หลายๆ ชั้น (Multiple levels of undo) เราจะเปลี่ยนจากการจำคำสั่งชิ้นล่าสุดชิ้นเดียว มาเป็นการเก็บรวบรวมคำสั่งไว้เป็นคลังประวัติศาสตร์ในลักษณะ List หรือ Stack พร้อมกับเก็บตัวชี้ตำแหน่งปัจจุบัน (Current pointer) ไว้ในตัวจัดการระบบ:

📄 ชื่อไฟล์: history_manager.gd (หรือ HistoryManager.cs)
GDScript
```python
class_name HistoryManager
extends Node

# ถังเก็บสะสมกล่องวัตถุคำสั่งเรียงตามลำดับเวลา
var _history_list: Array[Command] = []
var _current_index: int = -1

func execute_command(new_command: Command, actor: CharacterBody2D) -> void:
    # ถ้ามีการกด Undo ค้างไว้แล้วสั่งคำสั่งใหม่ ให้ตัดประวัติศาตร์อนาคตที่ค้างอยู่ทิ้งไป
    if _current_index < _history_list.size() - 1:
        _history_list = _history_list.slice(0, _current_index + 1)
        
    # รันคำสั่ง และเก็บกล่องคำสั่งลงถังประวัติ
    new_command.execute(actor)
    _history_list.append(new_command)
    _current_index = _history_list.size() - 1

func undo_action() -> void:
    if _current_index >= 0:
        var command: Command = _history_list[_current_index]
        command.undo() # ควักกล่องล่าสุดออกมาสั่งถอยหลังกลับอดีต
        _current_index -= 1

func redo_action(actor: CharacterBody2D) -> void:
    if _current_index < _history_list.size() - 1:
        _current_index += 1
        var command: Command = _history_list[_current_index]
        command.execute(actor) # สั่งกล่องเดิมเดินหน้าทำซ้ำอีกรอบ
```
มันต้องใช้ระเบียบวินัยในตัวเองค่อนข้างสูงในการกำกับควบคุมให้แน่ใจว่า "ทุกๆ การแก้ไขเปลี่ยนแปลงข้อมูลในเกมจะต้องวิ่งผ่านวัตถุ Command เสมอ" แต่ทันทีที่คุณทำตามกฎเหล็กนี้ได้สำเร็จ ส่วนที่เหลือของระบบจะเป็นเรื่องง่ายและทรงพลังมากครับ ตัวอย่างเช่น ระบบบันทึกคำสั่งแต่ละเฟรมไว้ใช้สำหรับรันระบบภาพรีเพลย์ย้อนดูการเล่น (Re-play) เป็นต้น

รูปแบบคลาสที่สวยหรู กับวิถีฟังก์ชันนอลที่ไร้สถานะ? (Classy and Dysfunctional?)
ในตอนต้น ผมบอกว่าคำสั่งคอมมานด์มีความคล้ายคลึงกับ First-class functions หรือ Closures (ฟังก์ชันที่หิ้วสถานะตัวแปรภายนอกไปกับตัวมันได้) แต่ตัวอย่างที่แสดงให้ดูเป็นการประกาศคลาสทั้งหมด นั่นเป็นเพราะภาษาดั้งเดิมอย่าง C++ มีระบบพวกนี้ค่อนข้างจำกัดและบริหารจัดการแรมยาก

ทว่า หากคุณใช้ภาษาที่มีระบบโครงสร้างยุคใหม่อย่าง JavaScript หรือแม้กระทั่ง GDScript ใน Godot 4 ที่มีระบบ Callable / Anonymous Functions คุณสามารถเลือกเขียนระบบคอมมานด์สไตล์ฟังก์ชันนอล (Functional Style) ที่สั้นกระชับได้ทันทีครับ เช่น การใช้คู่ฟังก์ชัน Closures ครอบสถานะตัวแปรแบบนี้:

GDScript
```python
# ตัวอย่างสไตล์ Functional ใน GDScript โดยใช้โครงสร้าง Dictionary อมฟังก์ชัน Callable
func make_move_unit_command(unit: Node2D, new_pos: Vector2) -> Dictionary:
    var old_pos: Vector2 # ตัวแปรนี้จะถูกหิ้วฝังติดไปกับฟังก์ชันภายในระบบความจำ (Closure)
    
    return {
        "execute": func():
            old_pos = unit.global_position
            unit.global_position = new_pos,
        "undo": func():
            unit.global_position = old_pos
    }
```
ถ้าคุณคุ้นเคยกับวิถีทางฟังก์ชันนอล วิธีการทำสิ่งต่างๆ แบบนี้จะรู้สึกเป็นธรรมชาติและสั้นกระชับมาก แต่ข้อดีของการเขียนในรูปแบบโครงสร้าง Class ชัดเจนตามตัวอย่างครึ่งแรกของบทเรียนคือ มันช่วยให้คนที่มาอ่านโค้ดในทีมสามารถเปิดดูตัวแปรฟาร์มข้อมูล (Fields) ได้ง่าย และระบุปฏิบัติการได้หลายมิติอย่างเป็นระเบียบมากกว่าครับ

ดูเพิ่มเติม (See Also)
การทำคลาสคำสั่งย่อยจำนวนมากในโปรเจกต์ อาจทำให้เกิดโค้ดซ้ำซ้อนขึ้นได้ แนะนำให้ประกาศคลาสฐานที่มีฟังก์ชันอำนวยความสะดวกระดับสูง (High-level methods) เตรียมไว้ให้คลาสลูกดึงไปรวมร่างเพื่อกำหนดพฤติกรรม สถาปัตยกรรมแบบนั้นจะเปลี่ยนฟังก์ชัน execute() ให้กลายเป็นแพทเทิร์นที่ชื่อว่า Subclass Sandbox Pattern ครับ

ในบางสถานการณ์ที่โครงสร้างวัตถุในเกมของคุณเรียงลำดับชั้นเป็นโครงสร้างต้นไม้ (Hierarchical Model) แบบ Node ใน Godot วัตถุชิ้นหนึ่งที่รับคำสั่งมาอาจจะส่งทอดคำสั่งปัดความรับผิดชอบไปให้วัตถุลูกน้องที่เป็นผู้ใต้บังคับบัญชาทำแทนได้ หากคุณวางระบบสไตล์นั้น คุณจะได้ผสมผสานเข้ากับแพทเทิร์นที่ชื่อว่า Chain of Responsibility Pattern ครับ

คำสั่งบางอันที่ไร้สถานะตัวแปรภายใน (Stateless) เช่น JumpCommand ในตัวอย่างแรกสุด การสร้างวัตถุชิ้นใหม่ทุกครั้งจะเปลืองแรมโดยใช่เหตุ แพทเทิร์นที่ชื่อว่า Flyweight Pattern จะก้าวเข้ามาช่วยจัดระเบียบเพื่อบริหารแชร์ใช้งานวัตถุชิ้นเดียวกันในจุดนี้ครับ
